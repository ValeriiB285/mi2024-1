Нижче — **типове завдання** (можна давати як екзаменаційне/практичне) на розробку **інформаційно-аналітичного дашборду обстрілів**. Дані — **навчальні/умовні**, **без реальних координат і назв позицій**.

---

## 1) Мета системи

Створити веб-дашборд, який показує **інтенсивність та структуру артилерійських обстрілів** (події/серії), тенденції в часі, розподіли за напрямками/типами, та список інцидентів з фільтрами.

Технології:

* **PostgreSQL** (англ. *PostgreSQL* /ˌpoʊstɡrɛsˌkjuːˈɛl/) — БД
* **Python** (англ. *Python* /ˈpaɪθɑːn/) — бекенд
* **Web + simple JS** (англ. *web* /wɛb/, *JavaScript* /ˈdʒɑːvəskrɪpt/) — фронтенд без фреймворків

---

## 2) Дані та предметна область (навчальна модель)

### Сутності

1. **Incident** (інцидент) — подія обстрілу
2. **Strike** (удар/епізод) — один запис у межах інциденту (опційно)
3. **LocationGrid** — умовний сектор/квадрат (тільки “GRID_###”, без географії)
4. **SourceType** — тип засобу (артилерія/міномет/РСЗВ тощо)
5. **Damage/Effect** — наслідки (без персональних даних)

---

## 3) Мінімальна схема БД (PostgreSQL)

### Таблиця `incidents`

* `id` (PK, UUID або SERIAL)
* `event_time` (timestamp, UTC)
* `sector_code` (text, напр. `GRID_042`)
* `direction` (text, напр. `N`, `NE`, `E`, `SE`, `S`, `SW`, `W`, `NW`)
* `source_type` (text: `ARTILLERY`, `MORTAR`, `MLRS`, `UAV_DROP`, `OTHER`)
* `caliber_mm` (int, nullable)
* `rounds_est` (int) — оцінка кількості
* `duration_min` (int, nullable)
* `intensity` (smallint 1..5) — умовна інтенсивність
* `target_type` (text: `POSITION`, `ROAD`, `LOGISTICS`, `UNKNOWN`)
* `effect_level` (smallint 0..3) — 0 без ефекту … 3 значний
* `notes` (text, nullable)

Індекси:

* по `event_time`
* по `sector_code`
* комбінований `event_time + direction + source_type`

### Наповнення

Згенерувати **не менше 1000 записів** за останні 90 днів (скриптом Python), щоб дашборд мав “живу” статистику.

---

## 4) Бекенд (Python + API)

Реалізувати REST API (наприклад, на FastAPI — англ. *FastAPI* /fæst eɪ piː aɪ/).

### Обов’язкові ендпоїнти

1. `GET /api/health` — статус сервісу
2. `GET /api/incidents`

   * параметри: `date_from`, `date_to`, `sector`, `direction`, `source_type`, `min_intensity`
   * пагінація: `page`, `page_size`
3. `GET /api/metrics/summary`

   * загалом інцидентів, середня інтенсивність, сумарні `rounds_est`, топ-напрямок, топ-сектор
4. `GET /api/metrics/timeseries?bucket=day|week`

   * кількість інцидентів по днях/тижнях
5. `GET /api/metrics/by_source`
6. `GET /api/metrics/by_direction`
7. `GET /api/metrics/heatmap`

   * матриця: `sector_code × week` (або `sector_code × day`) з кількостями

Вимоги до якості:

* валідація параметрів (дати, діапазони, enum-значення)
* захист від SQL-ін’єкцій (тільки параметризовані запити/ORM)
* CORS налаштований для фронтенду

---

## 5) Фронтенд (Web + simple JS)

Одна сторінка `index.html` + `app.js` + `styles.css` (мінімально).

### Компоненти дашборду

1. **Фільтри**: період, сектор, напрямок, тип засобу, мін. інтенсивність
2. **KPI-картки**: всього інцидентів, сумарні постріли, середня інтенсивність, топ-напрямок
3. **Графік тренду** (лінійний або стовпчиковий) по днях/тижнях

   * можна: Chart.js (англ. /tʃɑːrt dʒeɪ ɛs/) або SVG вручну
4. **Розподіли**: за напрямками та типами (2 діаграми)
5. **Heatmap** (таблична “теплова карта”) по секторах vs тижнях
6. **Таблиця інцидентів** з пагінацією та кнопкою “деталі” (модальне вікно)

UX-вимоги:

* кнопка “Застосувати фільтри”
* кнопка “Скинути”
* індикатор завантаження
* повідомлення про помилки API

---

## 6) Аналітика, яку має показувати система

Обов’язково:

* динаміка обстрілів у часі
* топ-5 секторів за кількістю інцидентів
* топ-напрямки
* структура за `source_type`
* середня інтенсивність по тижнях
* “аномалія”: день/тиждень, де значення > (середнє + 2σ) **або** > 150% від медіани (простий метод — на вибір)

---

## 7) Обмеження безпеки/етики (обов’язково у звіті)

* **не використовувати реальні координати/назви позицій/підрозділів**
* тільки умовні `sector_code` і синтетичні дані
* не зберігати персональні дані
* показати короткий розділ “як це переноситься на реальні дані” без деталей, що можуть зашкодити

---

## 9) Критерії оцінювання (швидка рубрика)

* БД + індекси + коректні типи: 20%
* API (фільтри, агрегації, якість відповідей): 30%
* Дашборд (візуалізації + UX): 30%
* Коректність аналітики/аномалій: 10%
* Документація/відтворюваність запуску: 10%
